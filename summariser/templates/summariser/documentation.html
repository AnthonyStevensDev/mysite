Smart Document Summariser

A full walkthrough of how this AI powered tool was designed, built, deployed, and debugged — including every problem encountered along the way and how it was solved.

1

Overview

The Smart Document Summariser is a web based AI tool that allows users to upload a document — either a PDF, Word file, or plain text file — and receive an instant AI generated summary along with five key points extracted from the content.

This was built as my first portfolio project to demonstrate real world AI integration using the Anthropic Claude API, file handling in Python, and full stack Django development. The goal was to build something genuinely useful rather than a tutorial clone.

The tool is built into my main portfolio site as a standalone Django app, styled to match the existing design system, and deployed live on Vercel.

Tech Stack

Python

Core programming language used for all backend logic, file processing, and API communication.

Django

Web framework used to handle routing, file uploads, template rendering, and the request response cycle.

Claude API (Anthropic)

AI model used to generate document summaries and key points from extracted text content.

Python library used to extract raw text from uploaded PDF files page by page.

python-docx

Python library used to extract text from Word documents paragraph by paragraph.

Tailwind CSS

Utility first CSS framework used to style the summariser page to match the existing portfolio design.

Project Architecture

The summariser was built as a separate Django app inside the existing portfolio project. This follows Django's separation of concerns principle — each distinct feature lives in its own self contained app.

Folder Structure 

summariser/
  templates/
    summariser/
      summariser.html      — upload form and results page
      documentation.html   — this page
  views.py                 — file extraction and API logic
  urls.py                  — URL routing for the app

Request Flow 

User uploads file
      ↓
Django receives file via request.FILES
      ↓
Python extracts text based on file type
      ↓
Text is sent to Claude API with a prompt
      ↓
Claude returns a summary and key points
      ↓
Django passes results to the HTML template
      ↓
User sees the summary on the page

Step by Step Build

01 Get the Anthropic API Key

Created an account at console.anthropic.com and generated an API key. The key was stored in the local .env file and added to Vercel's environment variables for production. It is never committed to GitHub.

02 Install Required Libraries

Three libraries were installed to handle AI communication and file reading: 

pip install anthropic PyPDF2 python-docx

After installing, requirements.txt was updated with pip freeze so Vercel could install the same packages during deployment.

03 Create the Summariser Django App

Rather than adding the tool to the existing myapp, a dedicated summariser app was created to keep concerns separated: 

python manage.py startapp summariser

The app was registered in settings.py under INSTALLED_APPS. This project uses three settings files — settings.py as a shared base, settings_dev.py for local development, and settings_prod.py for production. Both dev and prod files import everything from settings.py and only override what needs to be different, such as DEBUG and ALLOWED_HOSTS. This means adding an app to settings.py once makes it available in all environments automatically.

04 Write the Text Extraction Logic

Three separate functions were written in views.py — one for each file type. A dispatcher function then checks the file extension and routes to the correct extraction function. This pattern keeps the code clean and easy to debug.

05 Write the Claude API Call

The extracted text is trimmed to 10,000 characters to control API costs, then sent to Claude with a carefully structured prompt. The prompt gives Claude a role, specific instructions, and a strict output format so the response can be reliably parsed: 

SUMMARY:
Your summary here

KEY POINTS:
- Key point 1
- Key point 2

The response is then split on the KEY POINTS: label and each section is parsed separately into Python variables.

06 Connect URLs and Deploy

A urls.py file was created in the summariser app and connected to the main mysite/urls.py. The tool was tested locally first, then pushed to GitHub which triggered an automatic Vercel deployment.

Key Concepts Learned

request.FILES vs request.POST

Uploaded files are stored in request.FILES, not request.POST. request.POST only contains text data. The form must also include enctype="multipart/form-data" otherwise the file is never actually sent to Django.

CSRF Protection

Every POST form in Django requires a {% csrf_token %} tag. This protects against Cross Site Request Forgery attacks where a malicious site tricks a user's browser into submitting a form without their knowledge.

Prompt Engineering

The quality and consistency of Claude's output depends heavily on how the prompt is written. Giving Claude a clear role, specific constraints, and a strict output format produces reliable results that can be parsed programmatically.

Bytes vs Strings

When Django receives an uploaded file it stores the contents as raw bytes. These must be decoded into a Python string using decode('utf-8') before they can be processed as text.

Environment Variables

Sensitive values like API keys are stored in a .env file locally and in Vercel's environment variables in production. They are never committed to GitHub. The python-dotenv library loads the .env file automatically when the server starts.

Separation of Concerns

Each Django app should do one thing and do it well. Building the summariser as its own app rather than adding it to the main myapp keeps the codebase organised, maintainable, and scalable as more projects are added.

Problems & Solutions

Every problem below was encountered during the real build of this project. Debugging these issues was as valuable as writing the code itself.

Problem 1 — requirements.txt encoding error

Error 

Could not parse requirements.txt: Unexpected '?'

Cause

Running pip freeze > requirements.txt in Windows PowerShell saves the file as UTF-16 encoding. Vercel expects UTF-8 and cannot read the file, displaying an invisible byte character at the start.

Solution

Manually created a new requirements.txt file in VS Code and pasted the package list directly into it. VS Code saves files as UTF-8 by default, which Vercel can read correctly.

Problem 2 — Static files returning 404 on Vercel

Error 

404 — /static/myapp/style.css not found

Cause

The staticfiles/ folder was listed in .gitignore which meant it was never pushed to GitHub. Vercel had no static files to serve. Additionally WhiteNoise was not wrapped around the application in wsgi.py.

Solution

Three fixes were required. First, staticfiles/ was removed from .gitignore. Second, WhiteNoise was added to wsgi.py. Third, Git's cached ignore rules were cleared: 

git rm -r --cached staticfiles/
git add staticfiles/
git commit -m "Include staticfiles for Vercel"
git push
Problem 3 — Secret key exposed in Git diff

Problem

When moving the Django SECRET_KEY to an environment variable, Git showed the old key value in the diff before it was removed from settings.py.

Solution

A new secret key was generated immediately using Django's built in utility and the old key was considered compromised. The new key was stored in .env locally and in Vercel's environment variables. The .env file was confirmed to be in .gitignore.

Future Improvements

User Authentication

Add Django's built in user authentication system so only registered users can access the tool. Include a free tier with a monthly usage limit to control API costs.

Summary History

Store past summaries in a database so users can revisit previous documents without uploading them again.

Export Options

Allow users to download the summary and key points as a PDF or Word document directly from the results page.

Ask Questions About the Document

Add a chat interface below the summary so users can ask follow up questions about the document content using the same Claude API session.

Smarter Text Chunking

Replace the simple 10,000 character trim with intelligent chunking that splits large documents into sections, summarises each, and combines the results for a more accurate overall summary.
